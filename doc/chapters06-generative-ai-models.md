# 2. 什么是生成式 AI 模型

**Generative AI Model（生成式 AI 模型）** 能根据输入的提示和上下文生成新的内容。输出可以是文本、图像、代码、音频或它们的组合。

与传统分类模型相比：

```text
传统分类模型
输入：邮件正文
输出：垃圾邮件 / 正常邮件

生成式 AI 模型
输入：请将这封邮件总结为三条要点
输出：一段新生成的摘要
```

生成式模型通常不是简单地从训练数据中“查找原句”，而是根据学习到的模式预测接下来最合适的内容单元。

---

# 3. Foundation Models（基础模型）

**Foundation Model** 是在大量、多样化数据上进行预训练、可通过提示或进一步适配来支持多种任务的通用模型。

同一个基础模型可能用于：

* 问答与对话；
* 文本摘要、改写与翻译；
* 分类、提取和信息整理；
* 代码辅助；
* 图像或多模态理解与生成。

其关键特点是“先广泛预训练，再面向具体任务使用”。这降低了每个应用从零训练模型的成本，但不意味着无需评估：基础模型对某个行业、语言或高风险任务未必天然可靠。

---

# 4. Large Language Models（大型语言模型）

**Large Language Model（LLM）** 是擅长理解和生成自然语言的大型基础模型。它通过处理大量文本模式，能够完成对话、摘要、提取、翻译、推理式问答和内容起草等任务。

LLM 的典型交互是：

```text
System Prompt：定义角色、边界与输出规则
User Prompt：提出具体任务
Context：提供相关事实、文档或对话历史
        ↓
LLM Inference
        ↓
Generated Response
```

LLM 擅长语言，但输出不是天然等于事实。对于企业知识问答、医疗、金融或法律等场景，应提供受控上下文、来源验证和适当的人类审核。

---

# 5. Multimodal Models（多模态模型）

**Multimodal Model（多模态模型）** 可以处理或生成多种数据模态，例如文本与图像，有时还包括音频、视频。

典型任务：

```text
输入：产品图片 + “描述图片中的主要卖点”
输出：文字说明

输入：图表截图 + “总结趋势并指出异常值”
输出：分析文字
```

选择多模态模型的理由是任务需要跨越多种信息形式；若只需处理纯文本或固定视觉任务，则专用文本或视觉能力可能更简单、更经济。

---

# 6. Training 与 Inference

## 6.1 Training（训练）

训练是模型从大量数据中学习模式并调整内部参数的过程。基础模型训练通常需要大量计算资源和时间。

## 6.2 Inference（推理）

推理是将已训练的模型用于新输入、生成结果的过程。用户向聊天应用发送提示并获得回答，就是一次推理。

```text
大量训练数据 → Training → 已训练模型
用户提示与上下文 → Inference → 模型输出
```

考试记忆：**训练让模型学会能力；推理让模型把能力用于当前请求。**

---

# 7. Token、Context 与 Context Window

## 7.1 Token

模型处理文本时通常会把文本拆分为称为 **Token** 的单位。Token 不完全等同于中文字符或英文单词；它是模型使用的文本片段。

Token 会影响：

* 输入和输出可容纳的长度；
* 推理的延迟和成本；
* 应用是否需要先分段、总结或检索内容。

## 7.2 Context（上下文）

Context 是模型在一次推理中可参考的信息，包括系统提示、用户问题、对话历史、检索到的文档和其他输入。

更相关、更可靠的上下文通常能帮助模型给出更贴切的回答；不相关或矛盾的上下文则可能降低质量。

## 7.3 Context Window（上下文窗口）

Context Window 是模型一次能够处理的 Token 总量上限，通常包括输入和输出。

```text
System Prompt + User Prompt + Conversation History + Retrieved Content + Output
                         ≤ Context Window
```

如果内容超过窗口，需要采用截断、分段、摘要或检索等策略。较大的窗口有助于处理长内容，但仍应只提供与任务相关的信息，以控制成本、延迟和干扰。

---

# 8. 模型参数与输出行为

生成模型常提供参数以调整输出特征。AI-901 层面重点理解其方向性影响：

| 参数或设置 | 一般影响 |
| --- | --- |
| Temperature | 较低时更稳定、保守；较高时更多样、更具随机性 |
| Maximum output tokens | 限制可生成的输出长度 |
| System prompt | 规定角色、任务边界、格式和行为约束 |
| Context | 提供完成任务所需的事实和背景 |

例如，抽取固定字段或生成标准化摘要时，通常希望输出更稳定；创意文案构思则可以允许更多样性。参数不能保证事实正确，事实准确性仍依赖可靠数据、验证和应用设计。

---

# 9. 模型选择

选择生成式模型时，应从业务目标倒推，而不是默认选择最大模型。

| 考量 | 要问的问题 |
| --- | --- |
| 能力 | 需要文本、图像、代码、嵌入还是多模态能力？ |
| 质量 | 是否能在代表性样例上满足任务标准？ |
| 延迟 | 用户是否需要实时响应？ |
| 成本 | 请求量、Token 长度和预算是否可接受？ |
| 上下文 | 是否需要处理长文档或检索内容？ |
| 安全与合规 | 数据、区域、权限和输出风险如何控制？ |

建议流程：

```text
定义任务与衡量标准
      ↓
选择少量候选模型
      ↓
使用真实样例和安全样例评估
      ↓
比较质量、延迟与成本
      ↓
选择并持续监控
```

# 10. Grounding 与幻觉

生成式模型可能产生语法流畅但并不正确、没有依据或无法验证的内容，这类现象常被称为 **Hallucination（幻觉）**。

**Grounding** 是把模型回答建立在可控、相关的数据源或上下文上，例如经过权限检查的企业知识库。它可以减少无依据回答，但不能完全消除错误，因此仍需：

* 提供明确任务和输出格式；
* 只检索高质量、最新且有权限的数据；
* 要求模型在信息不足时说明不确定性；
* 让应用显示来源或供用户核验；
* 对高风险输出加入人工复核。

# 12. 易混淆概念

## Foundation Model vs LLM

* **Foundation Model**：可支持多类任务的广义预训练模型；
* **LLM**：以语言理解和生成见长的一类基础模型。

## Prompt vs Context

* **Prompt**：对模型发出的任务指令或问题；
* **Context**：模型完成任务时可参考的更广泛信息。

## Training vs Inference

* **Training**：从数据学习和更新模型；
* **Inference**：对新输入产生预测或生成结果。

## Fine-tuning vs Prompting

* **Prompting**：通过指令和上下文影响单次或多次模型输出；
* **Fine-tuning**：使用特定数据进一步调整模型。它通常成本和治理要求更高，并非所有场景都需要。
