# 2. 什么是 Microsoft Foundry

**Microsoft Foundry** 是用于构建和运营 AI 应用的平台体验。它将模型发现、试验、部署、评估、工具连接与应用开发能力集中起来，帮助团队更快将 AI 能力转化为可管理的解决方案。

可以把它理解为：

```text
业务需求
   ↓
选择模型与能力
   ↓
在项目中试验、评估和配置安全措施
   ↓
部署模型并连接应用、数据和工具
   ↓
监控、改进和治理
```

Foundry 并不意味着所有 AI 应用都不需要代码。它既支持在界面中快速试验，也支持通过 SDK 和 API 将能力集成到生产应用。

# 3. 核心概念

## 3.1 Foundry Project（项目）

项目是团队组织 AI 开发工作的边界。通常可以在项目中管理模型、连接、评估、部署、协作成员与相关资源。

项目的价值包括：

* 将同一应用的资源和配置聚合在一起；
* 为团队协作、访问控制和成本管理提供范围；
* 让开发、测试和运营活动更易追踪。

## 3.2 Model Catalog（模型目录）

模型目录用于浏览可用模型并了解其能力、用途、限制、输入输出方式和部署选项。

选择模型时不应只比较“模型是否最新”，而应结合：

* 任务类型：对话、摘要、视觉、嵌入、推理等；
* 输入与输出模态：文本、图像、音频或多模态；
* 质量、延迟、吞吐量和成本；
* 区域、合规和数据处理要求；
* 安全能力及应用场景风险。

## 3.3 Model Deployment（模型部署）

部署是把选定模型配置为应用可调用的终结点或服务实例的过程。模型在目录中“可发现”，不等于已经能被你的应用调用；通常需要先部署。

```text
Model Catalog
    ↓ 选择
Model
    ↓ 配置
Deployment
    ↓ 调用
Application / API / SDK
```

部署时通常需要考虑容量、区域、认证、访问权限、版本和成本。生产应用还应处理限流、重试、失败降级和日志记录。

## 3.4 Playground（试验场）

Playground 让开发者在无需先编写完整应用的情况下，快速尝试模型、提示词、参数和上下文，并观察输出质量。

适合用于：

* 比较不同模型的回答效果；
* 试验系统提示和用户提示；
* 调整温度等生成参数；
* 验证任务是否适合使用生成式 AI；
* 发现安全、格式或事实准确性问题。

Playground 的结果是原型验证，不应取代生产环境的系统测试、权限设计和持续监控。

## 3.5 Tools 与 Connections（工具和连接）

AI 应用常需连接企业数据、搜索能力或外部业务系统。工具和连接让模型或 Agent 能在受控的范围内检索信息、调用函数或执行任务。

例如：

```text
用户询问订单状态
    ↓
AI 应用理解问题
    ↓
受权限控制的订单查询工具
    ↓
返回真实订单数据
    ↓
模型组织成自然语言答复
```

应遵循最小权限原则：模型或应用只能访问完成任务所必需的数据和操作。

## 3.6 Foundry SDK

SDK 让开发者在代码中调用模型、管理应用交互并集成平台能力。它适合将原型变成可重复、可测试、可维护的应用流程。

在考试中，记住 SDK 的核心作用即可：**让应用程序以代码方式使用 Foundry 中的模型和相关能力。**

# 4. AI 应用开发生命周期

一个典型的生成式 AI 应用流程如下：

| 阶段 | 主要活动 | 关键问题 |
| --- | --- | --- |
| 定义 | 明确用户、任务与成功标准 | AI 是否适合这个任务？ |
| 选择 | 浏览模型并比较能力 | 哪个模型在质量、成本、延迟上合适？ |
| 原型 | 用 Playground 测试提示和样例 | 输出是否有用、稳定且安全？ |
| 构建 | 接入应用、数据和工具 | 如何认证、检索信息和处理失败？ |
| 评估 | 测试质量、安全与场景覆盖 | 是否满足业务和 Responsible AI 要求？ |
| 部署 | 发布可调用的模型与应用 | 如何控制访问、容量和版本？ |
| 运营 | 监控、收集反馈、迭代 | 环境变化后是否仍然可靠？ |

# 5. 从场景到能力选择

## 5.1 企业知识问答

需求：员工根据内部政策文档提问并获得有依据的回答。

可采用：生成式模型 + 受控知识检索 + 清晰的系统指令 + 引用或来源提示。重点是将回答建立在授权数据上，而不是让模型凭空猜测。

## 5.2 内容起草助手

需求：帮助市场团队起草邮件和初稿。

可采用：文本生成模型 + 提示模板 + 内容审阅流程。重点是保留人工编辑权，并为品牌、敏感信息和事实准确性设置护栏。

## 5.3 图像理解应用

需求：从产品照片中识别缺陷或生成描述。

可选择具有视觉或多模态能力的模型；若任务是固定字段或对象检测，也应评估专用视觉能力是否更合适。

# 6. Responsible AI 与安全实践

Foundry 中的开发活动应始终包含 Responsible AI 考量：

* 在设计前确定不可接受的输出与行为；
* 用代表性测试样例评估质量和群体差异；
* 对提示注入、越权工具调用和敏感信息泄露进行测试；
* 使用内容安全、身份认证、访问控制和日志；
* 对高影响决策提供人工监督、说明和申诉路径；
* 持续评估上线后的反馈、错误和模型漂移。

> 工具帮助控制风险，但组织仍需对使用场景和最终影响负责。

# 7. AI-901 考试重点

```text
Model Catalog  → 查找、比较并选择模型
Deployment     → 将模型配置为应用可调用的服务
Playground     → 快速试验模型、提示和参数
Project        → 组织资源、协作与开发活动
SDK            → 在代码中集成模型能力
Tools          → 让应用在受控范围内访问数据或执行操作
```

常见判断题思路：

* “先比较模型效果” → Model Catalog / Playground；
* “让应用能调用模型” → Deployment；
* “用代码整合到应用” → SDK；
* “调用业务系统中的实时数据” → Tools / Connections。

# 8. 易混淆概念

## Model vs Deployment

* **Model**：具备某种 AI 能力的模型本身；
* **Deployment**：该模型为特定应用配置的可调用实例或终结点。

## Playground vs Production Application

* **Playground**：快速试验和验证；
* **生产应用**：还需要认证、错误处理、监控、测试、成本和治理设计。

## Model Catalog vs SDK

* **Model Catalog**：发现和选择模型；
* **SDK**：从应用代码中使用模型和平台能力。
