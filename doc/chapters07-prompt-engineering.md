# 第 7 章 Prompt Engineering

> Prompt Engineering（提示工程）是设计、测试和改进提示词，使生成式 AI 更稳定地完成目标任务的方法。

---

# 1. 学习目标

学习后，你应能区分系统提示、用户提示与上下文；使用清晰的提示结构；理解 Grounding、输出格式和模型参数对结果的影响。

# 2. Prompt 的基本结构

```text
System Prompt：角色、边界、规则
User Prompt：具体任务
Context：完成任务所需的事实、文档或数据
Output Format：期望的格式、语言和长度
```

一个有效提示通常明确说明：任务、受众、输入、约束和输出格式。

```text
你是一名客服助理。
仅依据下方订单信息回答；信息不足时说明无法确认。
用中文输出三条要点，不要编造订单状态。
```

# 3. System Prompt 与 User Prompt

**System Prompt** 用于定义模型的总体行为，例如角色、语气、安全边界和输出规则。**User Prompt** 是用户当前提出的请求。

```text
System Prompt：你是企业政策助手；不得编造政策。
User Prompt：请说明员工差旅报销的金额上限。
```

系统提示不能代替应用层的权限与安全控制；不要把机密规则或用户敏感信息不加保护地放入提示中。

# 4. 提示优化原则

* **清晰具体**：避免“写得好一些”这类模糊要求；
* **提供上下文**：给出可靠且与任务相关的资料；
* **规定格式**：如 JSON、表格、标题或固定字段；
* **分解复杂任务**：将大任务拆成提取、判断、生成等步骤；
* **给出示例**：必要时展示输入和理想输出；
* **要求处理不确定性**：信息不足时回答“不足以判断”。

# 5. Grounding

Grounding 是将模型回答建立在受控的、相关的事实来源上，例如企业知识库、产品目录或经授权的文档。它有助于降低幻觉，但不能替代数据质量检查和人工审核。

```text
用户问题 → 检索授权文档 → 将相关片段作为 Context → 模型生成有依据的回答
```

# 6. 模型参数

| 设置 | 作用 |
| --- | --- |
| Temperature | 低值通常更稳定；高值通常更多样 |
| Maximum output tokens | 限制答案长度 |
| System prompt | 统一角色和行为边界 |
| Context | 为回答提供事实和背景 |

字段抽取、标准化摘要等任务通常偏好清晰指令和较低随机性；创意构思可以接受更多样性。

# 7. 评估与改进

不要只看一条“看起来不错”的回复。应使用代表性测试集，覆盖普通问题、边界问题、恶意输入、长上下文和信息不足的情形，评估准确性、格式遵从性、安全性、延迟与成本。

# 8. 易混淆概念

* Prompt 是指令；Context 是模型可参考的信息。
* Prompt Engineering 是优化交互方式，不等同于重新训练模型。
* Grounding 提供依据，不保证输出绝对正确。

# 9. 练习题

## 练习题 1

要求模型“只依据提供的政策文档回答，找不到时明确说明”，主要目的是：

A. 增加随机性  
B. Grounding 并降低无依据回答  
C. 训练新模型  
D. 执行 OCR

<details><summary>查看答案</summary>

**答案：B。** 受控上下文使回答有可依赖的事实来源。

</details>

## 练习题 2

希望输出始终为包含 `title`、`summary` 和 `actions` 的 JSON，最有效的提示改进是：

A. 明确规定输出格式和字段  
B. 删除所有上下文  
C. 增加随机性  
D. 改用对象检测

<details><summary>查看答案</summary>

**答案：A。** 明确格式约束能提高输出的一致性。

</details>

# 10. 一句话记忆

> **好提示要明确任务、上下文、约束和输出格式；Grounding 提供事实依据，评估确保提示在真实场景中可靠。**
